# RAAM Dataset Selection Research - 2026-07-04

This note records the evidence-backed dataset recommendation for the next RAAM-LM
training phases. It is based on 22 Exa searches, full dataset-card fetches for
the leading candidates, public Hugging Face Dataset Viewer metadata, SciSpace
paper search, and the current RAAM artifact state.

## Current RAAM Gap

The current promoted 84M checkpoint was trained on a curated-dominant mix:

- 2,880 curated records generated by this repo.
- 500 public chat records sampled from WildChat and OpenAssistant/oasst1.
- No meaningful real code/SWE trace slice in the selected mixed file.

The model can sometimes produce tiny memorized code patterns such as `is_even`,
but it fails on slightly larger functions, JSON/tool calls, and patches. The
next data must therefore prioritize verified small coding tasks, function bodies,
tests, and short patch/edit examples before long agent traces.

## Best Datasets By Role

### 1. Immediate Code SFT: `nvidia/OpenCodeInstruct`

Use this as the main public SFT source for the next coding ladder. It contains
about 5M coding question-answer pairs with domains, generation algorithms,
LLM-judgement metadata, unit tests, execution status, and average test score.
License: CC-BY-4.0.

Why it fits RAAM:

- It directly targets code generation rather than broad chat.
- Unit-test metadata lets us filter toward executable/correct examples.
- It is large enough to sample easy-to-medium examples for an 84M repair run.

Recommended filter:

- Python first.
- Prefer examples with passing tests and high `average_test_score`.
- Keep short samples under the current context limit.
- Downsample heavily; do not blindly train all 5M on the 84M model.

Source: https://huggingface.co/datasets/nvidia/OpenCodeInstruct

### 2. Function-Body Grounding: `Samip/Scotch`

Use this for docstring-to-function and function-context grounding. The card
describes about 19M functions from permissively licensed GitHub repositories;
about 4M have docstrings. License: MIT for the dataset, with permissive upstream
repo licenses.

Why it fits RAAM:

- The current model fails when moving from `is_even` to multi-line functions.
- Scotch gives many real function bodies, identifiers, docstrings, context, and
  license fields.
- It is much cleaner for learning function shape than random chat records.

Recommended filter:

- Python first, then JavaScript/TypeScript if needed from other sources.
- Functions between 3 and 80 lines.
- Strong docstring or simple signature.
- Exclude tests, generated files, minified files, and overly dependency-heavy code.

Source: https://huggingface.co/datasets/Samip/Scotch

### 3. Unit Test Skill: `KAKA22/CodeRM-UnitTest`

Use this to teach tests and execution thinking. It has about 77k Python unit-test
records derived from CodeFeedback/TACO with ground-truth code, generated
solutions, unit tests, FAR/FRR metrics, and Apache-2.0 dataset licensing.

Why it fits RAAM:

- Our next gate needs executable tests for `count_even`, `safe_int`,
  `parse_port`, etc.
- The dataset is explicitly about robust unit tests, not just code-shaped text.

Recommended filter:

- Prefer low FAR and low FRR tests.
- Convert into paired tasks: "write implementation", "write pytest", and
  "explain failing case".

Source: https://huggingface.co/datasets/KAKA22/CodeRM-UnitTest

### 4. Short Patch/Edit Skill: `bigcode/commitpackft`

Use this for simple edit/diff behavior. CommitPackFT has about 702k filtered
commit examples across 277 languages and includes old/new contents, commit
messages, language, license, and repo. License: MIT dataset, with sample-level
upstream licenses.

Why it fits RAAM:

- It teaches natural-language edit intent mapped to concrete code changes.
- It is more realistic than hand-written synthetic diffs but easier than full
  SWE agent trajectories.

Recommended filter:

- Single-file Python changes first.
- Small diffs, imperative commit subjects, permissive licenses.
- Convert old/new contents into unified diff SFT records.

Source: https://huggingface.co/datasets/bigcode/commitpackft

### 5. Agentic SWE Traces: `nvidia/Open-SWE-Traces`

Use after basic code generation is stable. The dataset has 207,489 trajectories
from OpenHands and SWE-agent, synthesized with Minimax-M2.5 and Qwen3.5-122B,
across real SWE-rebench-V2 issue statements with permissive repository licenses.
It includes trajectory roles, tools, resolved status, and reference/model patch
metadata. License: CC-BY-4.0 plus upstream MIT/Apache/BSD repo licenses.

Why it fits RAAM:

- This is the most directly aligned public dataset for autonomous coding-agent
  behavior: plan, tool use, repo context, patch, and verification.
- It includes resolved/unresolved labels, so we can train on high-quality subsets
  and use others for contrast later.

Recommended filter for first use:

- Python only.
- `resolved == 1`.
- OpenHands non-thinking traces first, then thinking traces if we choose to
  train reasoning text.
- Keep only short trajectories or distill them into smaller "state -> action"
  records until RAAM has longer context.

Source: https://huggingface.co/datasets/nvidia/Open-SWE-Traces

### 6. Additional Agentic SWE Traces: `nvidia/SWE-Zero-openhands-trajectories`

Use as the second agentic source. It contains 318,115 OpenHands trajectories over
118,092 issues, synthesized by Qwen3-Coder-480B-A35B-Instruct. License:
CC-BY-4.0 plus upstream MIT/Apache/BSD repo licenses.

Why it fits RAAM:

- It is larger than Open-SWE-Traces and explicitly SFT-oriented.
- It supplies issue statements, trajectories, final model patches, and source
  dataset labels.

Recommended filter:

- Same as Open-SWE-Traces: Python, short, patch-present, low-noise examples.
- Use after the model can already emit clean function bodies and simple patches.

Source: https://huggingface.co/datasets/nvidia/SWE-Zero-openhands-trajectories

### 7. Repo-Level Patch Tasks: `nebius/SWE-rebench-V2-PRs`

Use for repo-level edit examples before or alongside agent traces. It has
126,300 real-world pull-request samples across multiple languages with problem
statements, PR descriptions, gold patches, test patches, FAIL_TO_PASS /
PASS_TO_PASS fields, interface metadata, and repo license fields. License:
CC-BY-4.0 plus upstream repo licenses.

Why it fits RAAM:

- This is closer to "issue -> patch" than raw commit data.
- It includes tests and environment config fields, which helps build evals and
  training records.

Recommended filter:

- Python, one or two modified files, small line count, clear tests.
- Use gold patch tasks before full multi-turn agent trajectories.

Source: https://huggingface.co/datasets/nebius/SWE-rebench-V2-PRs

### 8. SWE Task Environments: `SWE-bench/SWE-smith`

Use later for executable task training/eval. Dataset metadata reports about 59k
rows; the README describes roughly 50k task instances from GitHub repositories
with executable environments. License: MIT.

Why it fits RAAM:

- It gives task instances, patches, fail-to-pass tests, and environments.
- It is excellent for training/evaluating agents once RAAM can interact with a
  harness.

Recommended use:

- Treat as a later phase. The current 84M model and 512-token context are not
  ready for full environment-scale tasks.

Source: https://huggingface.co/datasets/SWE-bench/SWE-smith

## Best Base / Continued-Pretraining Sources

### `HuggingFaceTB/smollm-corpus`

Best small-model-oriented base source. It includes `python-edu`, a Python subset
scored by an educational code model and extracted from Stack v2, plus
FineWeb-Edu-Dedup and Cosmopedia-v2. License: ODC-BY, with Stack v2 upstream
license considerations for Python-Edu.

Use:

- `python-edu` for Python syntax and educational code.
- `fineweb-edu-dedup` for general language only in a small ratio.

Source: https://huggingface.co/datasets/HuggingFaceTB/smollm-corpus

### `HuggingFaceTB/dclm-edu`

Good small-model web source, filtered for educational quality and used in
SmolLM2-135M/360M. License: CC-BY-4.0.

Use:

- Only as a small general-language stabilizer, not the main code source.
- Prefer `edu_int_score >= 3`.

Source: https://huggingface.co/datasets/HuggingFaceTB/dclm-edu

### `bigcode/starcoder2data-extras`

Strong code-adjacent source used in StarCoder2. Important subsets: documentation,
issues, StackOverflow, IR Python/Rust/C++/low-resource, Kaggle, OpenWebMath, and
LHQ. Dataset Viewer metadata reports about 51.6M rows across configs.

Use:

- `documentation`, `issues`, `stackoverflow`, and `ir_python` first.
- Avoid over-weighting contest/math subsets until we add contamination controls.

Source: https://huggingface.co/datasets/bigcode/starcoder2data-extras

### The Stack v2 / Stack v2 train variants

Best large-scale open code corpus for serious pretraining, but not the immediate
fix. It is huge, provenance-heavy, and license-sensitive. The Stack v2 card and
StarCoder2 paper describe a 67.5TB raw collection, 32.1TB deduplicated, and about
900B unique training tokens. It is the right source for a later larger model or
larger continued-pretraining phase, not for the next 84M repair run.

Use:

- Prefer already filtered smaller derivatives first: SmolLM `python-edu`,
  StarCoder2 extras, and language-specific samples.
- Keep provenance and attribution metadata.

Source: https://huggingface.co/datasets/bigcode/the-stack-v2

## Math / Reasoning Additions

### `nvidia/OpenMathInstruct-2`

Large math instruction dataset with about 14M problem-solution pairs and 1M/2M/5M
downsampled splits. License: CC-BY-4.0.

Use:

- 2-5% in code SFT or continued pretraining for reasoning.
- Filter long questions as recommended by the dataset card.

Source: https://huggingface.co/datasets/nvidia/OpenMathInstruct-2

### `AI-MO/NuminaMath-1.5`

High-quality post-training math dataset with 896,215 competition-level problems.
License: Apache-2.0.

Use:

- Prefer this over massive synthetic math when we need a small, clean math slice.

Source: https://huggingface.co/datasets/AI-MO/NuminaMath-1.5

## Datasets To Avoid As Primary Training Sources

- WildChat / OASST general chat: useful at low ratio for turn-taking, but our
  current run already showed that a chat slice does not teach coding.
- HumanEval, MBPP, BigCodeBench, LiveCodeBench, SWE-bench Verified: keep as eval
  or decontamination references, not training data.
- Raw The Stack v2 at this stage: too large and governance-heavy for the next
  repair run.
- Full long agent traces for 84M immediately: too long/noisy unless distilled or
  filtered into short action records.

## Recommended Training Mixes

### Next 84M Repair Run

Goal: make the model reliably write small and medium Python functions, stop
trailing into junk, and pass a coding ladder eval.

Use approximately:

- 35% repo-generated coding ladder data: `is_even`, `is_odd`, `count_even`,
  `filter_even`, `safe_int`, `parse_port`, simple tests, and simple diffs.
- 30% `nvidia/OpenCodeInstruct`, filtered to short Python examples with passing
  unit tests / high average score.
- 15% `Samip/Scotch`, filtered to Python docstring-function examples.
- 10% `KAKA22/CodeRM-UnitTest`, converted into implementation/test/debug tasks.
- 5% `bigcode/commitpackft`, filtered to small single-file Python diffs.
- 5% current curated RAAM behavior records to preserve chat/tool markers.

Do not include broad WildChat/OASST in this repair run except possibly a tiny
turn-format stabilizer. It was not the missing ingredient.

Implementation entrypoint:

```bash
python scripts/make_agentcoder_executable_sft.py \
  --output-dir runs/agentcoder_executable_sft \
  --opencode-jsonl /data/executable_sources/opencode.jsonl \
  --scotch-jsonl /data/executable_sources/scotch.jsonl \
  --coderm-unittest-jsonl /data/executable_sources/coderm_unittest.jsonl \
  --commitpackft-jsonl /data/executable_sources/commitpackft.jsonl \
  --ladder-repeats 4 \
  --curated-anchor-repeats 0
```

For direct Hugging Face sampling, add `--use-hf` and source-specific limits such
as `--opencode-limit 5000 --scotch-limit 2000 --coderm-unittest-limit 1000
--commitpackft-limit 1000`. The builder uses the `datasets` package when
available and otherwise falls back to the read-only Hugging Face Dataset Viewer
API. Defaults are `nvidia/OpenCodeInstruct` config `train`, `Samip/Scotch`
config `python`, `KAKA22/CodeRM-UnitTest` config `default`, and
`bigcode/commitpackft` config `python`.

Keep limits small until the manifest confirms source counts, filters, and
held-out eval cases are correct. On the July 4, 2026 smoke sample, CodeRM rows
were too large for the default `max_answer_chars` / `max_tests_chars` filter and
contributed no train records; that is acceptable for a first gate, but the next
larger repair run should either raise those limits selectively or add a
CodeRM-specific small-test extractor. The builder writes:

- `agentcoder_executable_train.jsonl`
- `agentcoder_executable_eval_cases.json`
- `agentcoder_executable_manifest.json`

The manifest records source counts, behavior/topic counts, filter settings, and
exact train/eval user-prompt overlaps. It intentionally does not execute
arbitrary public-source code during data preparation; executable eval cases are
created only when rows provide structured JSON `args`/`expected` tests.

### 150M-250M Code Foundation Run

Goal: stronger syntax, library familiarity, and function-level generalization.

Use approximately:

- 30% SmolLM `python-edu` / filtered Stack v2 Python derivatives.
- 20% Scotch functions with context and docstrings.
- 20% OpenCodeInstruct.
- 10% StarCoder2 extras: documentation, issues, StackOverflow, IR Python.
- 10% CommitPackFT / small diffs.
- 5% CodeRM-UnitTest.
- 5% NuminaMath or OpenMathInstruct-2.

### Agentic Coding Run

Goal: repo-aware issue solving, tool/action formatting, patches, and verification.

Use only after the model can write clean functions and patches:

- 30% Open-SWE-Traces, filtered to resolved/short/Python first.
- 25% SWE-Zero OpenHands trajectories, filtered to short patch-present examples.
- 20% SWE-rebench-V2-PRs, filtered to simple PRs with clear tests.
- 10% SWE-smith or R2E-Gym trajectories/tasks.
- 10% OpenCodeInstruct / CodeRM tests to preserve local code correctness.
- 5% RAAM curated tool/JSON/patch format anchors.

## Final Recommendation

For RAAM's current state, the "best dataset" is not a single public corpus. The
best immediate answer is:

1. Build our own small verified coding ladder for the exact behaviors the model
   is failing.
2. Mix in `nvidia/OpenCodeInstruct` as the main public SFT source.
3. Add `Samip/Scotch` for real function bodies and docstrings.
4. Add `KAKA22/CodeRM-UnitTest` for test-writing and executable validation.
5. Add `bigcode/commitpackft` for small edit/diff behavior.
6. Only after that, add `nvidia/Open-SWE-Traces`, `nvidia/SWE-Zero-openhands-trajectories`,
   and `nebius/SWE-rebench-V2-PRs` for true agentic coding.

This ordering directly addresses the observed failure mode: RAAM is not yet
missing broad chat; it is missing clean, repeated, verifiable code-generation
and edit data.
